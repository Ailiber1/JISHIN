# JISHIN_（地震避難シミュレーター）ゲーム仕様書

## コンセプト
**"Shake. Judge. Survive."** — **「その一秒の判断が、命を分ける。」**
地震発生から避難完了までをリアルに体験する3D防災シミュレーションゲーム。
オフィス・学校・自宅など現実の空間を舞台に、正しい避難行動を学びながらサバイバルする。
自治体・学校・企業の防災訓練としても活用できるシリアスゲーム。
**地方自治体の防災イベントや学校の防災教育で正式に採用されるレベルのクオリティを目指す。**

### 品質基準（自治体・教育機関で採用されるために）
```
本ゲームは「ブラウザゲーム」ではなく「防災教育ツール」として通用する品質を目標とする。

■ 情報の正確性
  - 全ての防災知識は公的機関（内閣府・消防庁・文部科学省）のガイドラインに準拠
  - 誤った情報は絶対に含めない（ガイドライン更新時は速やかに反映）
  - 参考資料の出典を明記（ゲーム内「防災知識」セクション）

■ 教育設計
  - 「体験→失敗→理解→再挑戦」のサイクルで自然に知識が身につく
  - エレベーター閉じ込め等、間違った行動を安全に体験できる
  - 結果画面で「なぜその行動が危険/正しいのか」を必ず解説
  - ステージごとの学習目標を明確に設定
  - 訓練モードで授業・研修の反復練習に対応

■ アクセシビリティ
  - URL共有だけで即プレイ可能（インストール不要）
  - PC・スマホ・タブレット全対応
  - 小学校低学年でも操作できるシンプルなUI
  - 高齢者でも読める文字サイズ（最小14px、推奨16px以上）
  - 日本語/英語切替対応
  - 色覚多様性に配慮（赤/緑だけでなく形・アイコンでも判別可能）

■ 視覚的クオリティ
  - PBRテクスチャによるリアルな3D空間（現実の学校・オフィスを忠実に再現）
  - 「ブラウザでこれが動くのか」と驚かれるレベルの描画品質
  - 自治体の防災イベント・学校の授業参観で見せても恥ずかしくない画面品質
  - スクリーンショットがそのまま広報資料・SNS投稿に使えるクオリティ

■ 信頼性
  - ゲーム内に監修情報を表示（参照ガイドライン一覧）
  - 「このゲームの防災知識は以下の公的資料に基づいています」と明記
  - 不適切な表現の排除（暴力的・差別的表現なし）
  - 教育目的に適さない要素の排除（広告なし・課金なし・完全無料）

■ 導入のしやすさ（自治体・学校向け）
  - 事前準備不要（URLをプロジェクターで映すだけ）
  - 1回のプレイ時間: 5〜10分（授業の1コマで複数ステージ体験可能）
  - 訓練モードで教師・講師が進行をコントロール可能
  - 結果画面を印刷/スクショして振り返り教材として使用可能
  - マルチプレイでクラス全員が同時体験可能
```

### 差別化ポイント
- ブラウザ3Dの地震避難ゲームは世界でほぼゼロ（完全な空白市場）
- KANKERI/MAZE-HUNTの建物内3D技術を流用（技術的再現が困難）
- リアルな落下物・揺れ・群衆パニックの物理演算
- PBRテクスチャによる現実世界に近い高品質3D地形
- 防災知識が自然に身につくゲーム設計
- URLを送るだけで即体験（アプリ不要）
- AIに「作って」と言って簡単に再現できないレベルの品質

## 1. ゲームの基本ルール

### ゲームの流れ
```
[0] 格言表示（タイトル画面表示前）
    防災に関する格言をフェードイン/アウトで表示
    3〜4秒間表示 → タイトル画面へ遷移
         |
[1] タイトル画面
    オレンジのマトリックス風デザイン
    モード選択 → ステージ選択
         |
[2] ブリーフィング（ステージ開始前）
    AIアドバイザーが場所の特徴と基本注意点を説明
    「このステージのポイント」を3つ提示
         |
[3] 日常パート（10〜15秒）
    地震発生前の通常状態
    プレイヤーは自由に歩き回り、周囲を確認できる
    ★ この間に「安全ゾーン」を事前に把握するのがコツ
         |
[4] 地震発生（緊急地震速報 → 揺れ開始）
    画面が揺れ、物が落下・転倒し始める
    プレイヤーは即座に「初期行動」を取る必要がある
    ├── 姿勢を低くする（しゃがみボタン）
    ├── 頭を守る（ガードボタン）
    ├── 安全な場所に移動する
    └── 危険な場所から離れる
    ★ AIアドバイザーがリアルタイムでアドバイス
         |
[5] 揺れ収束 → 避難判断フェーズ
    揺れが収まった後の行動を選択
    ├── 火の始末（キッチンがあるステージ）
    ├── ドア/窓を開けて避難経路確保
    ├── ブレーカーを落とす
    ├── 靴を履く（ガラス散乱時）
    └── 周囲の安否確認
    ★ 行動の順序と選択が採点に影響
         |
[6] 避難フェーズ
    建物から出て安全な場所まで避難する
    ├── エレベーターは使用可能だが乗ると途中停止→閉じ込め→ゲームオーバー
    │   ★ 間違った選択を「体験させる」ことで教育効果を高める
    ├── 正解: 非常階段で避難（余震で揺れる演出）
    ├── 落下物・倒壊物を避けながら移動
    ├── ブロック塀・自動販売機から離れる
    ├── NPCを助けながら避難（加点）
    └── 指定避難場所へ到着でクリア
         |
[7] 結果発表
    行動採点 + 防災知識の解説
    ├── スコア（100点満点）
    ├── 正しかった行動 / 間違った行動の振り返り
    ├── 「知っておくべきポイント」解説
    └── ランキング登録（ソロモード）
```

### 採点システム
```
[スコア計算（100点満点）]

■ 初期行動（30点）
  - 地震発生から3秒以内にしゃがむ        : +10点
  - 頭を守る行動を取った                  : +10点
  - 安全な場所（3原則を満たす場所）に移動  : +10点

■ 揺れ中の行動（20点）
  - 危険物（窓ガラス・棚・家電）から離れた  : +5点
  - 揺れ中に慌てて外に飛び出さなかった      : +5点
  - エレベーターを使用しなかった            : +5点
  - 落下物に当たらなかった                  : +5点

■ 揺れ後の行動（30点）
  - 火の始末を行った（該当ステージ）        : +6点
  - 避難経路を確保した（ドアを開ける）      : +6点
  - ブレーカーを落とした                    : +6点
  - 靴を履いた（ガラス散乱時）              : +6点
  - 周囲の安否確認を行った                  : +6点

■ 避難行動（20点）
  - 階段で避難した（エレベーター不使用）    : +5点
  - ブロック塀・自販機から離れて移動した    : +5点
  - 避難場所に到達した                      : +5点
  - 制限時間内に避難完了                    : +5点

■ 減点要素
  - 揺れ中に外に飛び出した                  : -10点
  - 落下物に当たった（1回につき）            : -5点
  - ガラス片の上を裸足で歩いた              : -5点
  - ブロック塀の近くを通過した              : -3点
  - 車で避難しようとした                    : -10点
  - 家族/仲間を置いて1人で逃げた            : -15点
  - パニックNPCに流されて危険行動           : -5点

■ ゲームオーバー条件（即終了 → 避難失敗）
  - エレベーターに乗る → 途中停止・閉じ込め → 避難失敗
  - ベランダから飛び降りようとする → 即ゲームオーバー
  - 落下物に3回被弾 → 「動けなくなりました」→ 避難失敗
  ※ ゲームオーバー後は必ず「なぜ危険なのか」の解説を表示
  ※ 教育目的のため「死亡」表現は使わない
     → 「閉じ込められた」「動けなくなった」「避難できなかった」と表現

■ ランク判定
  S : 90〜100点 「防災マスター」
  A : 75〜89点  「冷静な判断力」
  B : 60〜74点  「基本は押さえている」
  C : 40〜59点  「もう少し知識が必要」
  D : 0〜39点   「危険な行動が多い」
```

## 2. ステージ一覧（全5ステージ）

### Stage 1: 学校（教室）
```
[舞台] 日本の一般的な小中学校の教室（3階）
[時間帯] 昼間（授業中）
[難易度] ★☆☆☆☆（チュートリアル）

[環境要素]
- 机と椅子（30セット）
- 教卓・黒板・テレビモニター
- 窓ガラス（廊下側 + 校庭側）
- 蛍光灯（天井）
- ロッカー・掃除用具入れ
- 理科室の薬品棚（発展要素）

[落下・転倒物]
- 蛍光灯が落下
- テレビモニターが落下
- ロッカーが転倒
- 窓ガラスが飛散
- 本棚から教科書が散乱

[避難ルート]
教室 → 廊下 → 階段（3階→1階）→ 校庭（避難場所）

[学べること]
- 机の下に入り、脚を持って頭を守る
- 先生（NPC）の指示に従う重要性
- 廊下では窓ガラスから離れる
- 「おはしも」（押さない・走らない・しゃべらない・戻らない）
- 校庭の中央に集合（建物から離れる）

[NPC — 先生と生徒たち]
  ■ 先生NPC（1名・教卓付近）
    - 地震発生時「机の下に隠れなさい！」と指示を出す
    - 揺れ収束後「廊下に出て！走らないで！」と誘導
    - 避難中は先頭に立ち、校庭まで生徒を引率
    - プレイヤーが先生の指示に従うと加点
    - 先生の近くにいると安全ゾーン判定にボーナス

  ■ 生徒NPC（25〜28名・子どもモデル）
    - 冷静な生徒（70%）: 先生の指示通りに机の下→避難
    - パニック生徒（20%）: 泣いて動けない / 教室から飛び出す
    - 負傷生徒（10%）: 落下物に当たり動けない
      → プレイヤーが「インタラクト」で声をかける → 加点
    - 群衆として廊下・階段を移動（将棋倒しリスクの演出）

[ステージ特有の判定]
- 先生の指示に従って行動 → 加点
- パニック生徒に流されて教室から飛び出す → 減点
- 動けない生徒を助ける → 加点
- 「おはしも」を守って避難（走らない） → 加点
```

### Stage 2: オフィスビル（高層階）
```
[舞台] 都市部のオフィスビル 12階フロア
[時間帯] 午後（業務中）
[難易度] ★★☆☆☆

[環境要素]
- デスク・PC・モニター（20席）
- コピー機・プリンター（キャスター付き → 移動してくる）
- ファイリングキャビネット（背の高い書棚）
- ウォーターサーバー
- 天井パネル・照明
- エレベーターホール（エレベーター2基）
- 非常階段（2箇所）
- サーバールーム（ガラス壁）

[落下・転倒物]
- キャビネットが倒れる
- コピー機がスライド移動してくる
- 天井パネルが落下
- PC・モニターが落下
- ウォーターサーバー転倒（水が広がる）
- 蛍光灯が落下

[避難ルート]
  ● 正解: オフィスフロア → 非常階段（12階→1階）→ ビル前の広場
  × 不正解: エレベーター → 途中停止 → 閉じ込め → 避難失敗

[エレベータートラップ]
  ★ エレベーターは実際に乗れる（選択肢として用意する）
  - エレベーターのボタンを押す → ドアが開く
  - 乗り込む → ドアが閉まる → 数階降りたところで停電
  - エレベーター内で完全停止 → 照明が消える → 非常灯のみ
  - 画面に「エレベーターが停止しました」表示
  - 閉じ込め演出（5〜8秒間）:
    - 暗い密閉空間 + 余震で揺れる
    - AIアドバイザー（赤吹き出し・警告表情）:
      「地震の後にエレベーターに乗ってはいけません。
       余震や停電で閉じ込められる危険があります。
       必ず非常階段を使って避難しましょう。」
  - → 「避難失敗 — エレベーターに閉じ込められました」
  - → ゲームオーバー画面へ
  - 結果画面で「なぜエレベーターが危険なのか」を詳しく解説

[学べること]
- 「落ちてこない・倒れてこない・移動してこない」3原則
- エレベーターは絶対に使わない（体験して学ぶ）
- 高層階では長周期地震動で揺れが長く続く
- 非常階段の位置を日頃から確認する重要性
- ブレーカーを落としてから避難

[NPC — オフィスの同僚たち]
  ■ 上司NPC（1名）
    - 揺れ中「机の下に！」と指示
    - 揺れ後「非常階段から避難！エレベーターは使うな！」
    - 冷静に非常階段へ誘導する（正しい行動の手本）

  ■ 同僚NPC（15〜18名）
```

## 3. 防災3原則（ゲーム全体の核）

### 「落ちてこない・倒れてこない・移動してこない」

ゲーム内のすべてのステージで、この3原則に基づいた安全ゾーン判定を行う。

```
[安全ゾーン判定ロジック]

各ステージに「安全ゾーン」と「危険ゾーン」を事前に設定

■ 安全ゾーン（緑表示）
  - 丈夫な机の下（学校・オフィス）
  - 柱の近く（構造壁）
  - 玄関・廊下（家具がない場所）
  - トイレ（狭い空間 = 倒壊しにくい）

■ 危険ゾーン（赤表示）
  - 窓ガラスの近く
  - 背の高い家具の前
  - 照明の真下
  - エレベーター内
  - ブロック塀の近く
  - ベランダ

■ 判定方式
  プレイヤーの位置 → 最寄りの危険オブジェクトとの距離を計算
  → 安全ゾーン内: 被害なし
  → 危険ゾーン内: 落下物ヒット判定あり（確率 + 距離で変動）
```

## 10. 3D環境・物理演算

### 揺れの表現
```
[震度別の揺れパラメータ]

| 震度    | 揺れ幅(m) | 揺れ速度 | 持続時間 | 落下物数 | 転倒物数 |
|---------|----------|---------|---------|---------|---------|
| 震度5弱 | 0.02     | 中      | 15秒    | 少      | なし    |
| 震度5強 | 0.05     | 中〜速  | 20秒    | 中      | 少      |
| 震度6弱 | 0.10     | 速      | 25秒    | 多      | 中      |
| 震度6強 | 0.15     | 速      | 30秒    | 多      | 多      |
| 震度7   | 0.20     | 最速    | 40秒    | 最多    | 最多    |

[揺れの実装]
- カメラを正弦波 + ランダムノイズで振動
- 床・壁・天井のメッシュも微振動
- 物体は物理演算（簡易Rigid Body）で転倒・落下
  - 棚: 一定振動超えで倒壊トリガー
  - 食器: 棚から個別に落下（パーティクル的）
  - 照明: 天井から落下（ヒモが切れる演出）
  - コピー機: キャスターで滑って移動

[余震]
- メイン揺れの収束後、15〜30秒間隔で2〜3回の余震
- 余震の揺れは本震の30〜50%の強さ
- 避難中に余震が来る → 一時停止してしゃがむ判断が必要
```

### 落下物の当たり判定
```
[ダメージシステム]
- HP制ではない（死亡はしない）
- 落下物に当たると:
  1. 画面にヒットエフェクト（赤フラッシュ）
  2. 一時的に移動速度低下（3秒間）
  3. スコア減点（-5点/回）
  4. AIアドバイザーの警告

- 致命的な落下物（天井崩落・大型家具直撃）:
  1. 画面暗転 + 「意識を失いました」表示
  2. 5秒後に復帰（その場で起き上がる）
  3. スコア大幅減点（-15点）
  4. 3回被弾で「避難失敗」→ ゲームオーバー

※ 教育目的のため「死亡」表現は使わない
   → 「意識を失った」「動けなくなった」と表現
```

## 16. プレイヤーキャラクター仕様

### モデル方式
```
KANKERI方式のプロシージャル棒人間 + 服装パーツで役割を識別。
GLBモデルは使わず、Three.jsの基本ジオメトリのみで軽量に実現する。

■ 採用理由
  - GLBモデルは1体6〜21MB → Stage 5で40体以上のNPCが必要なため重すぎる
  - 棒人間+服装パーツなら1体あたり220〜240ポリゴン（数KB）→ NPC大量配置が可能
  - 防災ゲームはキャラの見た目よりも「環境のリアリティ」が重要
  - KANKERI/MAZE-HUNTで実証済みの描画パイプラインをそのまま流用
  - 服装パーツの追加は+10〜20ポリゴン/体 → パフォーマンスへの影響はほぼゼロ
```

## 17. カメラ仕様（TPS）

```
KANKERI方式のTPS（三人称視点）カメラ。

■ 基本パラメータ
  - カメラ距離: 5.0m（プレイヤーの背後）
  - カメラ高さ: 2.5m（プレイヤーの頭上やや後方）
  - FOV（視野角）: 60度
  - Near Clip: 0.1m / Far Clip: 200m

■ カメラ操作
  - PC: マウス移動で回転（感度調整可能）
  - モバイル: 右側スワイプで回転
  - 上下角度制限: -30度〜+60度（地面にめり込み防止）

■ 壁・障害物との衝突
  - カメラとプレイヤー間にRaycastを飛ばす
  - 障害物にヒット → カメラをプレイヤーに近づける（最小距離1.5m）
  - 障害物を抜けたら滑らかに元の距離に戻る（lerp係数0.1）
  - 室内（教室、オフィス、リビング）では自動で距離を3.5mに縮める

■ 地震時のカメラ演出
  - 揺れ中: カメラにも正弦波+ランダムノイズを加算
    - 震度5弱: 振幅0.01m / 震度7: 振幅0.08m
  - 落下物被弾時: カメラが一瞬ブレる（衝撃演出）
  - ゲームオーバー時: カメラがゆっくり引いていく演出

■ 周囲確認モード（Eキー / モバイルボタン）
  - カメラ距離を8mに引き、角度をやや俯瞰に
  - 移動不可（確認中は立ち止まる）
  - 日常パートで安全ゾーンを事前確認するのに使用
```

## 18. 3Dマップ規模・レイアウト

### ステージ別マップ寸法
```
■ Stage 1: 学校（教室 + 廊下 + 階段 + 校庭）
  教室: 9m × 8m（机30セット配置可能な標準サイズ）
  廊下: 幅3m × 長さ40m（教室4つ分）
  階段: 幅2.5m、踊り場あり（3階→1階 = 各階高さ3.5m）
  校庭: 30m × 30m（避難集合場所）
  合計プレイエリア: 約50m × 50m

■ Stage 2: オフィスビル（12階フロア + 非常階段 + 1階ロビー）
  オフィスフロア: 25m × 15m（デスク20席+会議室+サーバールーム）
  エレベーターホール: 5m × 4m（エレベーター2基）
  非常階段: 幅1.5m、踊り場あり（12階→1階 = 各階高さ3.2m）
```

### 24.1 スマホ縦横切替 レイアウト崩壊防止（KANKERI教訓 — 最重要）

```
★ KANKERIで多発中のバグ: PC画面は問題なくても、スマホの縦↔横回転で
  UIが重なる・ボタンが画面外に出る・モーダルがスクロールできない等が頻発。
  JISHIN_では設計段階からこれを根絶する。

■ 基本原則: 「縦持ち」と「横持ち」は別レイアウトとして設計する
  - 縦持ち（portrait）: 16:9以上の縦長。HUDは上部+下部に分離
  - 横持ち（landscape）: 画面高さが極端に小さい（350〜500px）ことを前提に設計
  - 「PCで問題ないからOK」は禁止。必ず両方向で実機検証する

■ CSS設計ルール（KANKERIの失敗から抽出）

  【ルール1】高さの単位
  - 全画面の高さ: 100dvh を使用（100vh はiOSでアドレスバー分ずれる）
  - フォールバック: height: 100vh; height: 100dvh;（dvh非対応ブラウザ用）
  - ゲーム画面: position:fixed + inset:0 で確実に全画面

  【ルール2】モーダル・パネルの高さ制限
  - max-height は必ず calc(100dvh - 余白) で指定
  - overflow-y: auto を必ず設定
  - 横持ちスマホ（max-height: 500px）では:
    → padding/margin を50%以上削減
    → font-size を80〜85%に縮小
    → 不要な装飾（スキャンライン、コーナー装飾等）を非表示
  - KANKERIの失敗: モーダルにmax-heightを付けず、横持ちで下半分が見切れた

  【ルール3】ゲーム中HUDの配置（最重要）
  - 絶対座標（top: 96px等）を使わない → 横持ちで画面外に飛ぶ
  - 代わりに以下のゾーン方式で管理:

    ┌─────────────────────────────────┐
    │  HUD上部ゾーン（top: 0〜60px）  │  ← タイマー、ステージ名
    │                                 │
    │         ゲーム3D描画             │
    │                                 │
    │  HUD左下ゾーン   HUD右下ゾーン  │  ← ジョイスティック / ボタン群
    └─────────────────────────────────┘

  - 各ゾーンは position:absolute + top/bottom/left/right で固定
  - ゾーン内要素は相対配置（gap/flexbox）で自動調整
  - KANKERIの失敗: ミニマップ・スタミナバー・情報バーが個別にtop値を持ち、
    横持ちで全部重なった

  【ルール4】ジョイスティック・ボタンの配置
  - ジョイスティック: 左下固定（bottom + left + safe-area）
  - アクションボタン群: 右下固定（bottom + right + safe-area）
  - 横持ちでもジョイスティックとボタンの位置関係は変えない
  - ボタンサイズ:
    - 縦持ち: 62px × 62px（通常）
    - 横持ち: 52px × 52px（少し縮小してゲーム視野を確保）
  - KANKERIの失敗: 横持ちでボタンが画面の半分を占めてゲームが見えなくなった

  【ルール5】orientationchange + resize の両方でレイアウト再計算
  window.addEventListener('resize', handleResize);
  window.addEventListener('orientationchange', () => {
    // orientationchange後にリサイズが遅延するブラウザがある
    setTimeout(handleResize, 100);
    setTimeout(handleResize, 300);
  });

  function handleResize() {
    const w = window.innerWidth;
    const h = window.innerHeight;
    const isLandscape = w > h;
    // 1. Three.js camera/renderer 更新
    camera.aspect = w / h;
    camera.updateProjectionMatrix();
    renderer.setSize(w, h);
    // 2. HUDレイアウト切替
    document.body.classList.toggle('landscape', isLandscape);
```

## 30. バグ予防 — 技術制約（過去ゲームの教訓から必須適用）

```
★ このセクションは KANKERI / MAZE-HUNT の開発で実際に多発したバグから
  抽出した「設計時点で守るべき技術ルール」である。
  開発者は各ステージ実装前にこのセクションを必ず読むこと。
```

### 30.1 壁・コリジョン設計（KANKERI全ステージで最頻出バグ）

```
■ 壁厚ルール
  - 壁の厚さ ≥ プレイヤー移動速度 × 2（1フレームで貫通しない厚さ）
  - 薄い壁（0.1〜0.2）は絶対禁止 → 最低0.5以上
  - 見た目が薄い壁でもコライダーは厚くする（可視メッシュ ± 0.1〜0.2）

■ コーナー処理
  - コーナー部分の壁は必ず「交差させて重ねる」（突き合わせ禁止）
  - 突き合わせ → 隙間 → すり抜け。これがKANKERIで最も多かったバグ

  NG（突き合わせ）:          OK（交差重ね）:
  ┌─────┐                   ┌──────┐
  │     │                   │      │
  └─────┘─────────           └──┬───┘
         └─────────              │
                                 └───────────

■ コリジョン判定の順序
  - 移動前に判定する（移動後に戻す方式は壁にめり込む原因）
  - X軸とZ軸は分離して判定する:
    1. X方向に仮移動 → 衝突判定 → 衝突したらX移動をキャンセル
    2. Z方向に仮移動 → 衝突判定 → 衝突したらZ移動をキャンセル
  - 同時に2壁に接触する場合（コーナー）は両方の壁を判定する

■ 建物間の隙間
  - 隣接する壁は 0.5以上重ねるか、見えない壁（transparent material）で隙間を塞ぐ

■ JISHIN_固有の注意点
  - Stage 1: 教室のドア開口部 → ドアが開いた状態のコリジョンを用意
  - Stage 2: ガラスパーティション → 飛散後はコリジョン削除（通過可能に）
  - Stage 5: 店舗のショーケース → 転倒後のコリジョン変更
```

### 30.2 多階建て・階段設計（KANKERI Stage4で修正10件以上）

```
★ JISHIN_は5ステージ中4ステージ（Stage 1,2,4,5）に階段がある。
  KANKERIのStage4階段だけで修正10件以上。同じバグを4ステージ分繰り返さないこと。

■ 各フロアのY座標を定数化（マジックナンバー禁止）
  const FLOOR_HEIGHT = {
    STAGE1: { FLOOR_1: 0, FLOOR_2: 3.5, FLOOR_3: 7.0, GROUND: -0.5 },
    STAGE2: { FLOOR_1: 0, FLOOR_2: 3.2, FLOOR_3: 6.4, ..., FLOOR_12: 35.2 },
    STAGE4: { FLOOR_1: 0, FLOOR_2: 3.0, ..., FLOOR_8: 21.0 },
  };
  全コードがこの定数を参照する。直値を書いた時点でバグの種。

■ 階段のprogress計算
  - 階段メッシュの回転方向とprogress方向を必ず一致させる
  - 回転行列で階段の両端Y座標を計算し、HIGH/LOWの位置を確認してからコードを書く
  - KANKERIの教訓: 回転方向とprogress方向が逆で、プレイヤーが浮遊・床下に沈んだ

■ 踊り場
  - 独立した床メッシュ + 独立したコリジョンを持つ
  - 階段と踊り場の接続部分は隙間なく接合（隙間 = 落下バグ）
  - 踊り場の外周には見えない壁（落下防止）

■ 階段室の入口/出口
  - 階段室の入口/出口を壁で塞がないこと（通行不可バグの最大原因）
  - 閉鎖壁を作る場合、必ず開口部のコリジョンを別に管理する

■ _onStairsフラグ
  - 「階段上」と「通常フロア」を明確に区別するフラグ
  - 階段上のY座標補正と通常フロアのY座標補正は別の定数で管理
  - 1Fと2Fの切替は境界Y座標（定数）で判定

■ 描画省略ルール（Stage 2: 12階、Stage 4: 8階）
  - 仕様書に「3〜4フロア分を実際に描画」とある
  - 省略フロアはトリガーゾーンで演出（カメラが暗転→フロアスキップ→明転）
  - 省略フロアにも壁コリジョンは必要（すり抜け防止）

■ 2F以上の落下防止
  - 全フロアの外周に見えない壁を配置（高さ2m以上）
  - 階段の手すりにもコリジョン（乗り越え防止）
```

### 30.3 エレベーター状態遷移（JISHIN_固有・新規ギミック）

```
★ エレベーターは過去ゲームにない完全新規ギミック。
  密閉空間 + カメラ + 状態遷移 + 物理演算 の組み合わせでバグが多発する。
  実装前にこの仕様を熟読すること。

■ エレベーターの状態遷移（ステートマシン）
  IDLE → DOOR_OPENING → DOOR_OPEN → PLAYER_ENTERING →
  DOOR_CLOSING → MOVING → EARTHQUAKE_STOP →
  POWER_OFF → TRAPPED → GAME_OVER

  各状態で許可されるアクション:
  | 状態 | プレイヤー移動 | カメラ操作 | インタラクト | 脱出可能 |
  |------|-------------|-----------|------------|---------|
  | IDLE | ○ | ○ | ボタン押下 | — |
  | DOOR_OPEN | ○ | ○ | 乗り込み | 外に出れる |
  | PLAYER_ENTERING | ×（自動移動）| ○ | × | × |
  | DOOR_CLOSING | × | ○ | × | × |
  | MOVING | × | ○（制限付き）| × | × |
  | EARTHQUAKE_STOP | × | ○（揺れ演出）| × | × |
  | POWER_OFF | × | ○（暗い）| 非常ボタン | × |
  | TRAPPED | × | ○ | × | × |

■ エレベーター内部空間のコリジョン
  - 内部サイズ: 2m × 1.5m × 2.5m（実際のエレベーターサイズ準拠）
  - 4面の壁 + 天井 + 床にコリジョンを配置
  - ドア面: DOOR_OPEN時はコリジョン消去、DOOR_CLOSING後にコリジョン復活
  - カメラとエレベーター内壁の最小距離: 0.3m（壁にめり込み防止）

■ カメラの制限（エレベーター内）
  - エレベーターに乗った瞬間にカメラ距離を 1.5m に縮小（通常3m）
  - 上下角度制限: -15度〜+45度（通常より狭く）
  - カメラがエレベーター外に出ないように Raycast でクランプ
  - POWER_OFF状態: 画面を90%暗くし、非常灯（赤い点滅ライト）のみ

■ 揺れ演出（EARTHQUAKE_STOP状態）
  - エレベーター全体をランダムに振動（振幅0.05m、周期100ms）
  - ガタン！の衝撃SE → 完全停止
  - カメラにも振動を加算（プレイヤーの恐怖感を演出）

■ ドアの物理
  - ドアはプレイヤーを押し出さない（挟まりバグ防止）
  - DOOR_CLOSING開始前にプレイヤーが完全に内部に入っているかチェック
  - プレイヤーがドア付近にいる場合 → 内部に押し込むフォースを適用
```

### 30.4 黒画面防止・描画初期化（KANKERI Stage1,4で多発）

```
■ ステージ生成関数のルール
  1. 引数で stageNum を必ず受け取る（グローバル変数に依存しない）
  2. 関数冒頭で全配列を初期化（cameraMeshes=[], colliders=[], npcs=[] 等）
  3. 素材定義（material）→ それを使うメッシュ生成 の順序を厳守
  4. try-catch で囲み、エラー時はフォールバックシーン（赤い床+白い箱）を表示

■ ライティング
  - MeshStandardMaterial 使用時は必ず AmbientLight + DirectionalLight を追加
  - 暗い場所（エレベーター内・停電シーン）でもemissive値を設定

  - ステージ固有のオブジェクト生成は if (stageNum === N) で囲む
  - 共通処理とステージ固有処理を SECTION コメントで明確に分離
```

### 30.5 ゲームループ・画面遷移のリソース管理

```
■ ゲームループ
  - requestAnimationFrame を使用（setInterval/setTimeout 禁止）
  - ループは常に1つだけ稼働（二重起動禁止）
  - ループ開始前: cancelAnimationFrame(animId) → animId = requestAnimationFrame(gameLoop)
  - deltaTime補正必須: dt = Math.min(rawDt, 50) / 16.667（50msクランプ）

■ 画面遷移時に必ず行うこと（チェックリスト）
  - [ ] cancelAnimationFrame(animId) でゲームループ停止
  - [ ] Firebase リスナーを全て off()
  - [ ] Three.js: geometry.dispose(), material.dispose(), texture.dispose()
  - [ ] scene.children を逆順で scene.remove() + dispose
  - [ ] renderer.dispose()（ステージ切替時のみ。同一ステージのリトライでは不要）
  - [ ] BGM停止: audio.pause(); audio.currentTime = 0;
  - [ ] ゲーム状態変数のリセット（スコア、タイマー、NPC配列、落下物配列 等）
  - [ ] HUDの表示リセット（前のステージの情報が残らないように）

■ Firebaseリスナーの管理
  - on() 登録前に必ず off() で既存リスナーを解除
  - リマッチ時にリスナーが蓄積する → メモリリーク → クラッシュ
  - 登録したリスナーの参照を配列で管理し、一括解除関数を用意:
    const listeners = [];
    function addListener(ref, event, callback) {
      ref.on(event, callback);
      listeners.push({ ref, event, callback });
    }
    function removeAllListeners() {
      listeners.forEach(l => l.ref.off(l.event, l.callback));
      listeners.length = 0;
    }
```

### 30.6 NPC AI・群衆の技術制約

```
■ AI移動の基本ルール
  - NPCにもプレイヤーと同じコリジョン判定を適用（省略しない）
  - 壁スライディング: 衝突時に壁の法線方向に速度を分解し、壁と平行な成分で移動
  - スタック検出: 2秒間の移動距離 < 0.5m → ランダム方向に転換
  - マップ外に出ないように毎フレーム位置をクランプ
  - AIスポーン位置は壁・家具から最低1m離す（埋まり防止）

■ ウェイポイント方式（階段・廊下・非常口）
  - 建物内のNPC移動はウェイポイント（通過必須ポイント）で制御
  - 直線移動で壁に衝突するルートを防ぐ
  - 各ステージのウェイポイントリスト（教室→廊下→階段→校庭 等）を定義

■ 群衆衝突（Stage 5: 地下街）
  ★ 50〜80体のNPC全対全判定は O(n²) → 性能崩壊する
  - 空間分割（グリッドベース）を使用:
    - マップを 5m × 5m のグリッドセルに分割
    - 各NPCを所属セルに登録
    - 衝突判定は同セル＋隣接セルのNPCのみ
    - 計算量: O(n²) → O(n × k)（k = セル内の平均NPC数）
  - NPCの押し合い解決:
    - 距離 < 1m のペアに斥力を適用
    - 斥力 = (1 - distance) × pushForce × 法線方向
    - pushForce は群衆密度に比例（密集するほど押し合いが強い）

■ パニック群衆の流れ
  - パニックNPCは「出口方向に最速で移動する」ルール
  - 出口付近で密集 → 将棋倒し判定:
    - セル内のNPC密度 > 閾値（1セルに8体以上）→ 転倒フラグ
    - 転倒NPC: 移動停止 + 倒れアニメーション + 周囲NPCの速度低下

■ NPC描画の最適化
  - 遠景NPC（カメラから20m以上）: ビルボード化（常にカメラを向く板ポリゴン）
  - 中景NPC（10〜20m）: LODモデル（服装パーツ省略、胴体のみ）
  - 近景NPC（10m以内）: フル描画（服装パーツ付き）
  - 画面外のNPC: 物理演算のみ（描画スキップ）
```

### 30.7 マルチプレイ同期の実装ルール

```
■ 位置同期
  - 送信: 100ms throttle（100msに1回だけ送信）
  - 座標: ワールド座標（x, y, z）をそのまま送信（画面サイズ非依存）
  - 回転: Y軸のみ送信（他軸は不要）
  - throttle実装:
    let lastSyncTime = 0;
    function syncPosition() {
      const now = performance.now();
      if (now - lastSyncTime < 100) return;
      lastSyncTime = now;
      ref.update({ x, y, z, ry, action, score, alive });
    }

■ 他プレイヤーの描画
  - 受信した座標への移動は線形補間（lerp）で滑らかにする
  - lerp係数: 0.15〜0.2（ガタガタしない + 遅延を感じない）
  - player.position.lerp(targetPosition, 0.15);

■ 地震イベントの同期
  - 地震の開始/終了はホストのタイムスタンプで統一
  - 各クライアントはホストのタイムスタンプを基準にローカルの揺れを開始
  - 落下物の位置はシード値で統一（全クライアント同じ落下パターン）

■ onDisconnect() の設定
  - ルーム参加時に必ず設定:
    playerRef.onDisconnect().remove();
  - ゲーム中にタブを閉じても確実にクリーンアップされる

■ リスナー解除のタイミング
  - ルーム退出時
  - ゲーム終了 → リザルト画面遷移時
  - リトライ時（古いリスナーが残ると二重発火）
  - タイトル画面に戻る時
```

### 30.8 スマホ操作の実装ルール

```
■ タッチイベント
  - click イベントは使わない（300msの遅延がある）
  - touchstart / touchend + e.preventDefault() を使用
  - ジョイスティック: touchstart → touchmove → touchend で制御

■ スクロール・ズーム防止（ゲーム画面全体）
  document.addEventListener('touchmove', e => e.preventDefault(), { passive: false });
  document.addEventListener('gesturestart', e => e.preventDefault());
  document.addEventListener('gesturechange', e => e.preventDefault());

■ viewport設定（必須）
  <meta name="viewport" content="width=device-width, initial-scale=1.0, user-scalable=no, viewport-fit=cover">

■ iOS Safariの固有問題
  - 100vh がアドレスバー分ずれる → 100dvh を使用（フォールバック付き）
  - ダブルタップでズームされる → touch-action: manipulation を全要素に適用
  - ゴムバンドスクロール → overscroll-behavior: none
  - 長押しでコンテキストメニュー → -webkit-touch-callout: none
```

### 30.9 KANKERIバグ実績データ（161件の修正から抽出した追加ルール）

```
★ KANKERI開発で実際に発生した修正コミット161件を分類した結果:
  - スマホUI/レイアウト崩壊: 37件（23%）← 最多カテゴリ
  - 壁・コリジョン・地形・階段: 50件（31%）← 最多カテゴリ
  - 黒画面・照明・fog: 27件（17%）
  - マルチプレイ: 20件（12%）
  - その他（操作・iOS・AI等）: 27件（17%）

  以下は、セクション30.1〜30.8でまだカバーされていないパターン。

■ 照明の「明→暗フラッシュ」問題（KANKERI修正7件）
  - 夜間ステージで、シーン初期化時のデフォルト照明（白い光）が一瞬表示され、
    その後に暗い照明に切り替わるため「ピカッ」とフラッシュが発生した
  - JISHIN_ではStage 3（一軒家・夜間想定なし）以外は日中ステージだが、
    エレベーター内の停電・地下街の停電シーンで同じ問題が起きる
  - 対策: シーン生成の最初のフレームから最終照明値を適用する
    → フェードオーバーレイや遅延切替は使わない
    → scene.background / scene.fog / light.intensity を生成関数内で即設定

■ fog密度と視認性のバランス（KANKERI修正5件）
  - FogExp2の密度が高すぎて画面が真っ暗 → 何度も微調整を繰り返した
  - JISHIN_のステージ別fog推奨値:
    | ステージ | fog密度 | 理由 |
    |---------|---------|------|
    | Stage 1: 学校 | 0.005 | 校庭が広い。視野を確保 |
    | Stage 2: オフィス | 0.008 | 室内。やや閉鎖感 |
    | Stage 3: 一軒家 | 0.006 | 中規模。自然な距離感 |
    | Stage 4: マンション | 0.007 | 廊下が長い。適度な奥行き感 |
    | Stage 5: 地下街 | 0.010 | 地下の閉鎖感。停電時は0.025に増加 |
  - 停電演出: fog密度を上げるのではなく、AmbientLight.intensity を下げる
    （fogを上げすぎると何も見えなくなる）

■ cameraMeshes配列の登録漏れ（KANKERI修正3件）
  - TPS（三人称）カメラはプレイヤーとカメラの間にRaycastを飛ばし、
    障害物にヒットしたらカメラを手前に引く
  - cameraMeshes に壁メッシュを登録し忘れると、カメラが壁を貫通して壁の裏が見える
  - 対策: 壁・床・天井・家具のメッシュを生成したら、必ず同じ行でcameraMeshesにpush
    const wall = new THREE.Mesh(geo, mat);
    scene.add(wall);
    colliders.push(wall);      // コリジョン用
    cameraMeshes.push(wall);   // カメラ貫通防止用 ← これを忘れがち

■ CSS詳細度の競合（!important地獄）（KANKERI修正4件）
  - メディアクエリ内のスタイルが、通常のスタイルに負けて適用されない
  - 何度も!importantを付けて対処 → コードが汚くなり、さらに競合が発生
  - JISHIN_の対策:
    1. CSSの記述順序を厳守: 基本スタイル → PC用 → タブレット用 → スマホ縦 → スマホ横
    2. メディアクエリ内では基本スタイルより詳細度を上げるセレクタを使用:
       body.portrait #hud-minimap { ... }  ← bodyクラスで詳細度を上げる
    3. !important は最終手段。使う場合はコメントで理由を書く

■ 再戦（リマッチ）時のレースコンディション（KANKERI修正8件）
  - リマッチボタン押下→ルーム状態変更→ゲーム再開始 の間にタイミングずれ
  - Player2がリマッチ処理中にルーム状態がRESULTからLOBBYに変わり、
    タイトル画面に戻されるバグが何度も発生
  - JISHIN_の対策:
    1. リマッチ処理中はフラグ（isRematchInProgress）で二重実行を防止
    2. ルーム状態の変更は必ずホストのみが行う（ゲスト側は状態変更を監視するだけ）
    3. 全プレイヤーの投票完了を確認してから状態遷移（タイムアウト20秒）
    4. cleanupMultiplayer() でFirebaseプレイヤーデータを削除しない
       → 削除すると再戦時に自分のデータが消えて参加できなくなる（KANKERI最大の教訓）

■ ゴーストルーム/ゴーストプレイヤー（KANKERI修正3件）
  - プレイヤー0人のルームが残り続け、新規ルーム作成時にIDが衝突
  - undefined HOST（ホストが切断したがonDisconnectが発火しなかった）
  - JISHIN_の対策:
    1. ルーム参加時にプレイヤー数を確認 → 0人ならルームを削除して新規作成
    2. ルーム一覧取得時に最終更新から10分以上経過したルームを自動削除
    3. ホストのdeviceIdが空/undefinedのルームは無効として扱う

■ iOS Safari BGM問題（KANKERI修正1件だが影響大）
  - iOS Safariでは audio.volume = 0 が無効（音量制御がOSに委譲される）
  - BGM OFF設定にしても音楽が流れ続けるバグ
  - 対策: audio.volume ではなく audio.muted = true を使用
    function setBGM(enabled) {
      bgm.muted = !enabled;  // volume=0 は iOS で効かない
    }

■ ジョイスティック反転バグ（KANKERI修正2件）
  - タッチ座標の計算で、ジョイスティック領域のoffsetを引き忘れて方向が逆になった
  - 横持ちに切り替えるとoffsetが変わり、別の方向にずれた
  - 対策: getBoundingClientRect() で毎回ジョイスティックの位置を取得
    → 固定offsetを使わない（回転で変わるため）
    const rect = joystickEl.getBoundingClientRect();
    const dx = touch.clientX - (rect.left + rect.width / 2);
    const dy = touch.clientY - (rect.top + rect.height / 2);

■ stageNum未定義エラー（KANKERI修正3件）
  - ステージ生成関数でstageNumをグローバル変数として参照していた
  - 画面遷移のタイミングでグローバル変数がリセットされ、undefinedになった
  - 対策: セクション30.4の通り、stageNumは必ず引数で渡す。
    追加ルール: stageNumを参照する全関数の冒頭で
    if (typeof stageNum !== 'number' || stageNum < 1 || stageNum > 5) {
      console.error('Invalid stageNum:', stageNum);
      return;
    }
    のバリデーションを入れる

■ ホスト専用UIが非ホストに表示される（KANKERI修正2件）
  - 缶の配置UIがゲスト側にも表示され、操作すると状態が壊れた
  - JISHIN_の対策:
    - ホスト専用UI（ステージ選択・震度選択・開始ボタン）には data-host-only 属性を付与
    - 非ホストにはグレーアウト + 「ホストが設定中...」メッセージ
    - ゲーム開始のトリガーはホストのFirebase書き込みのみ（非ホストは状態変更を検知してゲーム開始）
```
